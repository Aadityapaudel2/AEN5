# A Structured Two-Model Reasoning Harness for Mathematical Problem Solving on Linux GPU Infrastructure

## Abstract

This paper presents the design, migration, and protocol refinement of a local large-language-model system for mathematical reasoning, centered on a Linux-based deployment environment and a structured two-model dialogue evaluator. The system was migrated from a Windows-first workflow to a remote Ubuntu H100 environment in order to improve reproducibility, runtime stability, and inference throughput. Two complementary interaction regimes were developed: a single-model interface for direct experimentation and a two-model evaluator for bounded solver-verifier reasoning. The canonical model family remained Qwen 3.5, with 4B serving as the canonical lightweight baseline and 9B explored as a stronger but heavier inference option.

The main contribution is a protocol shift away from open-ended conversational reasoning and toward a bounded mathematical harness in which one model acts as a solver and another acts as a verifier. The protocol enforces an explicit progression through planning, challenge, solving, verification, revision, and reverification. This structure was introduced in response to observed failure modes in free-form loops, including identity drift, premature solving, weak critique, excessive runtime, and unbounded conversational continuation after apparent success. The refined evaluator also preserves loop state across resumptions, supports side-specific system-level prompt injection, and terminates only when the same solver final answer has been verified twice consecutively.

The resulting system is best understood not as a chatbot but as a local reasoning scaffold for mathematical problem solving. Early practical evidence suggests that protocol quality materially affects performance: in cases where a solo model can fail, a disciplined two-body exchange can recover a correct answer. The work therefore supports a central thesis: for difficult mathematical tasks, structured interaction design may be as important as raw model scale.

## 1. Introduction

Recent local language-model systems are often evaluated as if better reasoning arises primarily from scaling model parameters or increasing output length. In practice, however, reasoning failures frequently originate earlier: from poor decomposition, premature commitment to a bad strategy, weak internal checking, and the absence of a disciplined verification loop. These issues are especially pronounced in Olympiad-style mathematics and similarly structured reasoning domains, where the distinction between “starting well” and “finishing cleanly” is decisive.

This project began with a practical engineering goal: bootstrap a Windows-developed local model environment on a Linux GPU instance and validate inference speed and protocol behavior there. The broader research direction quickly became clearer. Rather than merely making local inference work on Linux, the objective shifted toward building a rigorous mathematical reasoning harness. The key insight was that the strongest advantage of multi-step reasoning does not come from generating more text, but from delaying commitment to a solution until the structure of the problem has been understood and contested.

This led to two interlocking efforts. First, the canonical single-model stack, AEN5, was adapted for Linux and stabilized across Tk and Qt-based user interfaces. Second, the two-model evaluator was transformed from a conversational demo into a bounded solver-verifier protocol. The system is intended not just to “solve problems,” but to create conditions under which a solution is more likely to be correct.

## 2. Motivation

The motivation for the two-model protocol came from recurring empirical issues:

1. A single model, even at 9B scale, can fail hard mathematical problems despite producing fluent reasoning.
2. Free-form two-model dialogue can still fail if the interaction is not constrained.
3. Unstructured loops often waste tokens on politeness, repetition, or identity drift rather than actual mathematical progress.
4. Verifier models tend to underperform when framed only as passive critics rather than active collaborators.
5. Resume behavior and targeted injection are essential evaluator features; without them, iterative analysis becomes cumbersome and brittle.

A stronger protocol was therefore necessary. The intended system had to preserve context across iterations, permit targeted evaluator steering, and enforce explicit reasoning phases. The verifier also had to become more useful: not merely rejecting flawed answers, but helping the solver correct them.

## 3. System Setting

### 3.1 Infrastructure

The system was migrated from a local Windows development context to a remote Ubuntu Linux environment backed by an NVIDIA H100 GPU. This migration served several purposes:

- faster and more stable inference for local checkpoints,
- easier package management for GPU-bound libraries,
- better support for reproducible benchmarking,
- a cleaner path for future Linux-native workflows.

The environment included:
- a Python virtual environment dedicated to the project,
- local model directories for Qwen checkpoints,
- the canonical AEN5 repository for single-model interaction,
- a dedicated Two_LLM_Model_Evaluator repository for structured two-model reasoning.

### 3.2 Canonical Model Choice

Qwen 3.5 4B remained the canonical baseline due to its lower overhead and easier iteration cost. Qwen 3.5 9B was supported as a stronger runtime option when GPU memory and latency budgets allowed. The central position taken during development was that protocol quality and evaluator structure were likely to matter more than simply moving to larger models for this class of task.

## 4. Migration and Linux Stabilization

### 4.1 AEN5 Single-Model Path

The single-model interface was adapted for Linux in both Tk and Qt variants. Several issues had to be addressed:

- Windows-oriented interpreter assumptions,
- Qt/XRDP/WebEngine instability on remote desktop sessions,
- streamer timeout failures with larger models,
- launcher reliability in a Linux virtual environment.

The Tk path was stabilized by removing the overly aggressive first-token streamer timeout that caused spurious queue timeout failures on slower 9B generation startup. The Qt path was hardened by reducing dependence on unstable WebEngine/OpenGL behavior in XRDP contexts and adding a plain-Qt rendering fallback more suitable for Linux remote sessions.

### 4.2 Two-Model Evaluator Path

The two-model evaluator was also made Linux-usable through:
- portable path handling,
- Linux example configs,
- headless benchmark support for SSH workflows,
- GUI launch support from desktop sessions,
- portable profile loading and generation configuration.

This migration was not merely infrastructural. It created the conditions for protocol iteration at speed.

## 5. Initial Evaluator Failure Modes

Before protocol redesign, several significant problems were observed.

### 5.1 Context Loss on Continue Loop

A completed loop would often behave like a fresh conversation when resumed, even when the user intended to continue the previous reasoning state. This undermined iterative mathematical analysis, where one often wants to revisit a hidden assumption or challenge a particular step after the main pass has ended.

### 5.2 Weak Prompt Injection Semantics

Side-specific prompt injection was unreliable. Injected instructions could be treated as ordinary chat content rather than evaluator-level steering, and could fail to influence the intended side’s next response in a controlled way.

### 5.3 Symmetric Solver-Verifier Collapse

Despite the existence of separate side names, solver and verifier could drift into near-identical behavior. In practice this meant the verifier sometimes became a second solver, reducing the system’s ability to identify flaws or course-correct.

### 5.4 Premature Stop Conditions

Earlier versions could terminate too early, for example on the first apparent acceptance condition, or after a fixed phase completion that did not reflect actual protocol success. This was unacceptable for mathematical reasoning, where a first “looks good” judgment is often not enough.

### 5.5 Open-Ended Chatter

Without a bounded protocol, the models could continue speaking after a usable solution had already been reached, wasting tokens and time.

## 6. Protocol Design Principles

The redesigned evaluator was guided by several principles.

### 6.1 Planning Before Solving

The system enforces a pause before solving. This is the central reasoning thesis of the project. The point is not simply to produce more thoughts, but to separate:
- understanding the problem,
- evaluating candidate plans,
- executing one plan,
- checking the result against criticism.

### 6.2 Distinct Functional Roles

The two sides must not merely differ in name. They must differ in function:
- Solver: advances the argument, chooses plans, constructs the derivation.
- Verifier: challenges assumptions, answers targeted questions, proposes corrective suggestions, and judges whether the current answer is justified.

### 6.3 Structured, Bounded Exchange

Every phase has a task and a constrained output shape. This discourages rambling and keeps the interaction focused on mathematical progress.

### 6.4 Resume and Injection as First-Class Features

A useful evaluator is interactive, not just batch-like. Continue Loop and side-specific injection were therefore treated as core workflow primitives, not incidental extras.

## 7. Final Protocol

The current evaluator protocol is no longer a generic chat loop. It is a bounded mathematical solver-verifier harness.

### 7.1 Phase 1: Solver Plan

The solver must:
- extract givens,
- state the target,
- propose candidate approaches,
- choose a plan,
- state a first concrete step,
- ask one concise question for the verifier if needed.

No full solution is allowed in this phase.

### 7.2 Phase 2: Verifier Challenge

The verifier must:
- critique the plan,
- identify weak assumptions,
- point out missing cases,
- call out invalid reductions or unsupported leaps,
- answer the solver’s most important question,
- give short concrete suggestions,
- ask the shortest correction-focused question that exposes the highest-risk flaw.

The verifier is not supposed to perform a full independent derivation.

### 7.3 Phase 3: Solver Solution

The solver must:
- execute the plan,
- incorporate verifier feedback,
- derive the result concretely,
- state a final answer.

### 7.4 Phase 4: Verifier Check

The verifier must:
- evaluate the current final answer,
- determine whether it is justified,
- either mark it `VERIFIED` or request `REVISE`,
- answer remaining solver questions,
- offer concise strengthening suggestions even in the verified case.

### 7.5 Phase 5: Revision and Reverification

If revision is requested:
- the solver revises or reaffirms the solution and restates the final answer,
- the verifier checks again.

This can repeat in bounded form.

## 8. Stop Condition

A key design correction was changing the termination rule.

The loop does **not** stop on the first verified instance.

Instead, termination requires:
1. the solver produces a concrete final answer,
2. the verifier marks that answer `VERIFIED`,
3. the solver later produces the same final answer again,
4. the verifier marks that same answer `VERIFIED` again.

Only then is the exchange considered protocol-complete. This acts as a lightweight confirmation pass and reduces the chance of a fragile one-shot acceptance ending the dialogue too early.

## 9. Continue Loop and Injection Semantics

### 9.1 Continue Loop

The evaluator now preserves:
- transcript state,
- role names,
- per-side histories,
- system prompts,
- prior loop artifacts.

If a user pastes a new prompt after completion and clicks Continue Loop, that prompt is treated as a controller continuation layered onto the existing discussion rather than a fresh session reset.

### 9.2 Injection

Prompt injection is now:
- side-specific,
- evaluator/system-level rather than ordinary dialogue,
- queued for the next turn of the targeted side,
- preserved across resumable completed loops,
- visible in the UI through queued counts on injection controls.

This makes targeted steering workable during mathematical evaluation.

## 10. Verifier Design

A major conceptual change was reframing the verifier as a collaborator rather than a purely adversarial critic.

The verifier is still responsible for:
- unjustified assumptions,
- missing cases,
- invalid reductions,
- arithmetic mistakes,
- logical gaps,
- unsupported conclusions,
- premature problem collapse.

But the verifier is also now expected to:
- answer the solver’s direct questions,
- provide short concrete suggestions,
- help the solver obtain a correct proof.

This matters because a good verifier is not simply one that says “no.” A good verifier helps surface what should change.

## 11. Performance Discipline

The evaluator was designed to remain bounded:
- constrained phases,
- turn limit controls,
- no endless success chatter,
- protocol-complete stop once the final answer is doubly verified.

Generation defaults were also made more practical for Linux use, favoring manageable token budgets over unreasonably large defaults that create latency and memory costs without proportional reasoning benefits.

## 12. Practical Engineering Outcomes

The Linux work produced several concrete engineering outcomes:

- stable virtual-environment-based runtime,
- usable local Qwen 3.5 checkpoint loading,
- validated single-model and two-model inference on H100,
- Linux-stable Tk single-model path,
- more reliable Qt single-model path with safer renderer behavior,
- Linux-ready evaluator copy support,
- structured protocol enforcement in the evaluator,
- resumable loop state,
- side-specific queued injection.

The work also established a reusable Linux branch workflow so future boots into Linux can begin from a clean baseline instead of reconstructing the environment from scratch.

## 13. Interpretation

This project supports a broader methodological claim: reasoning quality is strongly shaped by protocol design.

Bigger models help, but they do not automatically solve:
- bad initial path selection,
- weak verification,
- conversational drift,
- unbounded output waste.

A disciplined solver-verifier exchange can in some cases recover from failures that defeat a solo model. This is especially relevant for hard mathematics, where correctness depends on structure, not just fluency.

## 14. Limitations

Several limitations remain.

1. The current evidence is empirical and engineering-driven rather than a formal benchmark suite.
2. Live GUI validation of every protocol edge case is always more limited than direct code-level enforcement.
3. The verifier still depends on the base model’s competence; protocol cannot create mathematical insight from nothing.
4. The two-pass verified stop condition is a heuristic for robustness, not a proof of correctness.
5. Fast-path attention optimizations were not the central focus of the final protocol work and may still leave performance headroom untapped.

## 15. Future Work

Several next directions are especially promising.

### 15.1 Benchmarking
Run controlled comparisons across:
- single-model direct solving,
- unstructured two-model dialogue,
- structured solver-verifier protocol.

### 15.2 Process-SFT
The best future tuning target may be process supervision rather than more direct-answer supervision:
- how to plan,
- how to question,
- how to verify,
- how to revise.

### 15.3 Better Artifact Tracking
The evaluator could explicitly store:
- chosen plan artifact,
- verifier objections,
- accepted revision state,
- final verified answer object.

### 15.4 Formal Evaluation Harness
The current system is close to a mathematical research harness. It would benefit from:
- batched test case runners,
- correctness labels,
- trace classification,
- protocol-ablation experiments.

## 16. Conclusion

The work described here transformed a local model setup into a Linux-based mathematical reasoning environment and converted a conversational two-model demo into a structured solver-verifier harness. The central insight is that mathematical reasoning quality depends not only on model size, but on protocol architecture. By enforcing planning before solving, distinct solver and verifier roles, resumable dialogue state, side-specific injection, and a stricter verification stop condition, the evaluator becomes meaningfully better aligned with the requirements of hard mathematical problem solving.

The system is therefore not just a UI around local models. It is an attempt to operationalize a stronger theory of local reasoning: that correctness emerges more reliably when generation is scaffolded by explicit structure, adversarial-but-helpful verification, and bounded iterative refinement.

Important implementation direction for the two-model reasoning system:

We are no longer aiming for “just solve the problem” behavior.

We want a full root-level mathematical reasoning protocol where the models must:

1. evaluate the givens
2. identify exactly what is being asked
3. generate candidate approaches
4. question and challenge those approaches
5. create a concrete gameplan
6. agree on the gameplan
7. only then produce the full solution
8. finally verify the solution against the agreed plan

Core principle:
The critical intelligence advantage is the pause before writing the first actual solution step.
Top solvers separate themselves by dissecting the problem before committing to a path.
A bad initial path often leads to a wrong final solution even if later algebra is clean.

So the architecture should enforce:
- no full solution until a plan artifact exists
- no open-ended rambling
- bounded phases
- explicit structure

Desired protocol shape:
Phase 1: Extract
- What are the givens?
- What is the exact target?

Phase 2: Explore
- What are 2-4 viable approaches?
- Which are weak or risky?
- Which seem strongest?

Phase 3: Commit
- Select one plan
- State why it is the chosen plan
- State the first concrete step

Phase 4: Solve
- Execute the chosen plan carefully

Phase 5: Verify
- Check whether the solution actually matches the agreed plan
- Check arithmetic / logic / hidden assumptions
- Flag any deviation or unsupported leap

Implementation priority:
This should be enforced structurally in prompts / protocol / loop logic, not left to the model’s free-form behavior.

The current issue:
The models can drift, solve too early, or talk without building a disciplined plan.
We want to change that.

Please implement the reasoning protocol accordingly.

Implement the next evaluator protocol upgrade in the Linux repo.

Goal
Make the two-model dialogue evaluator behave like a bounded mathematical reasoning protocol rather than an open-ended chat loop.

Required behavior

1. Structured reasoning flow
The models should behave as:
- Solver
- Verifier

The intended sequence is:

Phase 1:
- Solver extracts givens, target, candidate approaches, and a chosen plan.

Phase 2:
- Verifier challenges the plan, weak assumptions, missing cases, invalid reductions, arithmetic risks, and unsupported jumps.

Phase 3:
- Solver produces an actual solution and a clear final answer.

Phase 4:
- Verifier either:
  - requests revision, or
  - marks the answer verified.

Phase 5:
- If revision was requested, Solver revises and gives a new final answer.
- Verifier checks again.

2. Stop condition
Do NOT stop immediately on the first accept / pass / revise message.

The stop condition should be protocol-complete:
- Solver has produced a concrete final answer.
- Verifier has explicitly marked the current answer as verified.
- The verified answer is the current final answer at the end of the cycle.
- Then stop.

Equivalent intended closeout:
- Solver gives final answer.
- Verifier evaluates.
- Solver gives final answer again if needed after revision.
- Verifier says VERIFIED.
- Then the loop stops.

3. Continue Loop semantics
Continue Loop must preserve the full prior conversation state.
If a loop ended and I paste a new prompt and click Continue Loop:
- it must continue from the existing dialogue state
- not start a fresh loop
- prior transcript, role names, system prompts, and per-side histories must remain intact
- only explicit reset/clear should wipe state

4. Prompt injection
Prompt injection is currently unreliable and must be fixed.
Requirements:
- side-specific injection
- treated as evaluator/system-level steering, not ordinary chat
- preserved history
- queued for the next turn of the targeted side
- if loop is complete but resumable, injection must apply on resume
- UI should clearly show which side has a queued injected instruction

5. Verifier prompt quality
Do NOT over-specialize the verifier to one domain like “hidden degrees of freedom”.
Keep the verifier universal across algebra, combinatorics, geometry, number theory, and logic.

Verifier should focus on:
- unjustified assumptions
- missing cases
- invalid reductions
- arithmetic mistakes
- logical gaps
- unsupported conclusions
- collapsing the problem too early

Verifier should ask the shortest correction-focused question that exposes the most dangerous flaw.

6. Performance discipline
Keep the loop bounded.
No endless “thank you / verified / ready for next problem” chatter after success.
Once the protocol-complete stop condition is met, terminate the loop cleanly.

Why this matters
We now have evidence that:
- 9B solo can fail on hard math problems
- 2-body can recover the correct answer
So the protocol itself matters.
The evaluator must now become a rigorous math harness, not a conversational demo.
