# MiamiOH Google Pilot Architecture

## Purpose

This note records the current pilot architecture for shipping Athena V5 to the MiamiOH MTH025C course without blocking on Canvas OAuth.

The pilot is intentionally narrow:

- Google provides identity
- the Canvas export provides broad course structure
- the at-a-glance guide provides authoritative date/schedule facts
- Athena V5 acts as a course-aware learning partner inside the AEN portal

## Identity

The live pilot path is Google-only.

Google users are treated as MiamiOH pilot users when:

- the Google email is verified
- the email ends with `@miamioh.edu`
- if present, the Google hosted-domain claim is `miamioh.edu`

This means the pilot does **not** depend on Canvas developer keys or Canvas OAuth to ship.

## Course Knowledge Layers

### 1. Static course export

The IMSCC export remains the broad course-content source for:

- module structure
- pages
- assignment names
- policy/support content
- general course framing

The bundle is generated by:

- `miamioh/build_canvas_bundle.py`

Key outputs:

- `institutions/miamioh/courses/250433/derived/course.json`
- `institutions/miamioh/courses/250433/derived/modules.json`
- `institutions/miamioh/courses/250433/derived/assignments.json`
- `institutions/miamioh/courses/250433/derived/content_chunks.jsonl`

### 2. At-a-glance override layer

The at-a-glance guide is the authoritative schedule layer for the pilot.

It is used for:

- quizzes
- exams
- final-exam window
- semester-date facts
- short policy reminders

Key output:

- `institutions/miamioh/courses/250433/pilot/pilot_overrides.json`

This layer should win over the export whenever the question is date-sensitive.

### 3. Pilot people layer

Instructor identity is now resolved through a dedicated pilot people file.

Key output:

- `institutions/miamioh/courses/250433/pilot/pilot_people.json`

Current behavior:

- `build_canvas_bundle.py` seeds the instructor of record from the at-a-glance guide
- the signed-in Google name is matched against that instructor record
- if it matches, Athena treats the user as the instructor
- otherwise the current pilot defaults the user to student

This is the clean extension point for a later roster upload.

## Prompt and Retrieval Behavior

The portal now combines:

- authenticated profile context
- curriculum context
- at-a-glance summary lines
- retrieved pilot override chunks
- retrieved export-bundle chunks

Key behavior:

- schedule/date questions should be answered from the at-a-glance layer first
- dates should be copied exactly as written in the course guide
- Athena should not imply live Canvas sync or personal gradebook access
- self-identity questions can use authenticated session metadata

## Instructor Recognition

The earlier pilot path treated all MiamiOH Google users as students. That is no longer acceptable.

Current path:

- role resolution runs during Google-pilot bootstrap
- instructor recognition is data-driven from course materials
- the authenticated profile is injected into the system prompt so Athena can answer:
  - who the user is
  - what role the user holds in the pilot course

- `Help me study for Quiz 6`

## Remaining Ship Gate

Handover is **not complete** until the live host is verified.

Required live check:

1. restart the deployed portal
2. fully sign out
3. sign back in with a real `@miamioh.edu` Google account
4. verify:
   - `What is this course about?`
   - `When is Exam 2?`
   - `When is the final?`
   - `What should I know about discussions?`
   - `Help me study for Quiz 6`
   - `What is my name and what is my position?`

Only after that live sign-in smoke passes should this pilot be marked ready for students.
